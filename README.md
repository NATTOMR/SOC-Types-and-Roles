## üîç What is a SOC?

A **Security Operations Center (SOC)** is a centralized facility where the information security team continuously monitors, analyzes, and responds to cybersecurity threats.

The primary purpose of a SOC is to:
- Detect security incidents  
- Analyze threats and vulnerabilities  
- Respond to and remediate incidents  
- Protect organizational assets using **people, processes, and technology**

---

## üèóÔ∏è Types of SOC Models

Depending on organizational needs, maturity, and budget, SOCs can be implemented in several models:

![SOC Models](https://github.com/NATTOMR/SOC-Types-and-Roles/blob/main/soc-models.png)

### üè¢ In-House SOC
- Built and managed internally by the organization  
- Requires a dedicated budget and skilled security staff  
- Provides full control over security operations  

### üåê Virtual SOC
- No permanent physical facility  
- Operates remotely across multiple locations  
- Flexible and cost-effective  

### ü§ù Co-Managed SOC
- Internal SOC team works alongside a Managed Security Service Provider (MSSP)  
- Shared responsibilities  
- Strong coordination is essential  

### üß≠ Command SOC
- Central SOC that oversees multiple smaller or regional SOCs  
- Common in large enterprises, telecom providers, and defense agencies  

---

## ‚öôÔ∏è People, Process, and Technology

A successful SOC relies on strong alignment between **people, processes, and technology**.

### üë• People
- Highly trained security professionals  
- Familiar with alerts, attack techniques, and incident response  
- Able to adapt quickly to evolving threat landscapes  
- Willing to continuously research and learn  

### üìã Processes
- Aligned with security frameworks and regulations such as **NIST, PCI-DSS, and HIPAA**  
- Standardized procedures to ensure consistent incident handling  
- Clearly defined escalation and response workflows  

### üß∞ Technology
- Tools for detection, prevention, investigation, and response  
- Includes SIEM, EDR, SOAR, and penetration testing tools  
- Tool selection should consider **organizational needs and budget**, not just market popularity  

---

## üë• SOC Roles

### üîé SOC Analyst
- Categorized into **Level 1, Level 2, and Level 3**  
- Monitors and investigates security alerts  
- Identifies root causes  
- Recommends remediation actions  

### üö® Incident Responder
- Performs initial assessment of security incidents  
- Focuses on containment, eradication, and recovery  

### üïµÔ∏è Threat Hunter
- Proactively searches for hidden or advanced threats  
- Investigates APTs and evasive attacks  
- Uses manual and automated techniques  
- Eliminates threats before business impact occurs  

### üõ†Ô∏è Security Engineer
- Maintains SOC infrastructure and security tools  
- Integrates SIEM with SOAR platforms  
- Ensures optimal performance of detection systems  

### üìä SOC Manager
- Oversees SOC strategy and operations  
- Handles budgeting, staffing, and performance metrics  
- Focuses on operational and managerial responsibilities

# ‚öôÔ∏è SIEM and Analyst Relationship

This lesson explains what **SIEM** is, why it is used in a **Security Operations Center (SOC)**, and how it supports the daily work of a **SOC Analyst**.

---

## üîç What is SIEM?

**SIEM (Security Information and Event Management)** is a security solution that collects, correlates, and analyzes logs and events in real time across an environment.

The primary goal of SIEM is to:
- Centralize event logging  
- Detect suspicious activity  
- Generate alerts for potential security threats  

Although SIEM platforms provide many advanced features, **SOC analysts primarily focus on alerts generated by SIEM rules and correlations**.

---

## üö® Example SIEM Alert

Consider the following scenario:

- A user on a Windows system enters **20 incorrect passwords within 10 seconds**
- This behavior is abnormal and likely malicious
- A SIEM rule is created to detect excessive failed login attempts within a short time window
- When the threshold is exceeded, the SIEM generates an alert

This allows SOC analysts to quickly identify and investigate suspicious authentication activity.

![SIEM Alert Example](https://github.com/NATTOMR/SOC-Types-and-Roles/blob/main/alert.png)

---

## üß∞ Popular SIEM Solutions

- Splunk

## üë®‚Äçüíª Relationship Between SOC Analysts and SIEM

Although SIEM platforms offer many capabilities, **SOC analysts mainly interact with alerts**.

- SIEM rules and correlations are typically created by **engineering or SIEM teams**
- Alerts generated by these rules are reviewed first by SOC analysts
- The analyst determines whether an alert is:
  - ‚úÖ A real security incident  
  - ‚ùå A false positive  

This process marks the **starting point of incident investigation in a SOC**.

## üîé Alert Investigation Workflow

From the SIEM dashboard, analysts can see multiple alerts generated from different data sources.

## üí° Quick Tip: Handling False Positives

False positives are unavoidable in SIEM environments.

A skilled SOC analyst:
- Identifies false alerts  
- Documents findings  
- Provides feedback to the SIEM team  
- Helps optimize detection rules  

### Example False Positive
- A SIEM rule flags URLs containing the word **‚Äúunion‚Äù** to detect SQL injection
- A user searches:  
  `https://www.google.com/search?q=sql+union+usage`
- An alert is generated, but no malicious activity is present

Sharing this feedback helps refine rules and reduce noise.

# üìú Log Management

As a SOC Analyst, **log analysis** is a core responsibility. Understanding how to use **Log Management systems** is critical for effective investigations.

---

## üîç What is Log Management?

**Log Management** is the process of collecting, storing, and searching logs from multiple sources in a centralized location.

Common log sources include:
- Web servers  
- Operating systems  
- Firewalls  
- Proxies  
- EDR solutions  

Centralized logging improves efficiency and reduces investigation time.

---

## ‚öôÔ∏è Why Centralized Logs Matter

Without log management:
- Analysts must query multiple systems individually
- This increases investigation time and error risk

With log management:
- All logs are accessible from a single interface
- One query can search across multiple data sources

In LetsDefend, log sources such as **Proxy, Exchange, and Firewall** are collected under a single system.

![Log Management Sources](https://github.com/NATTOMR/SOC-Types-and-Roles/blob/main/log-1.png)

---

## üéØ Purpose of Log Management for SOC Analysts

SOC analysts commonly use log management to:
- Check communication with suspicious IPs or domains
- Identify lateral movement
- Detect data exfiltration
- Validate SIEM alerts  

### Example 1: Malware Investigation
- Malware communicates with `letsdefend.io`
- Analyst searches for `letsdefend.io` in Log Management
- Determines whether any internal devices contacted the C2 server

### Example 2: Data Exfiltration Check
- SIEM alerts that a host is leaking data to `122[.]194[.]229[.]59`
- The device is isolated and remediated
- Analyst searches logs to verify if **any other systems** contacted the same IP

This step ensures no additional compromised hosts exist.

---

Log Management allows analysts to query logs quickly and efficiently:

# üß™ EDR ‚Äì Endpoint Detection and Response

A **SOC Analyst** spends a significant amount of time using **EDR** while investigating endpoint devices. This lesson explains what EDR is, why it is important for SOC operations, and how analysts use it effectively during investigations.

---

## üîç What is EDR?

**Endpoint Detection and Response (EDR)**, also known as **Endpoint Threat Detection and Response (ETDR)**, is an endpoint security solution that provides:

- Continuous, real-time monitoring of endpoints  
- Collection of endpoint telemetry data  
- Detection using rules and behavioral analysis  
- Automated response and investigation capabilities  

> **Definition source:** mcafee.com

EDR enables SOC analysts to gain deep visibility into endpoint activity and respond quickly to threats.

---

## üîé Analysis with EDR

Some commonly used EDR solutions in enterprise environments include:
- Carbon Black  
- SentinelOne  
- FireEye HX

- Review running processes  
- Inspect suspicious files

---

## üöß Containment

When an endpoint is compromised, **containment** is a critical step.

### Why isolate a device?
- Prevent the attacker from maintaining access  
- Stop lateral movement within the internal network  

EDR solutions provide a **containment feature** that:
- Isolates the device from internal and external networks  
- Allows communication only with the EDR management console  
- Enables continued investigation while preventing further damage  

![Endpoint Containment](https://github.com/NATTOMR/SOC-Types-and-Roles/blob/main/EDR-6.png)

The device remains isolated until:
- Vulnerabilities are fixed  
- Malicious activity is removed  
- The system is safe to return to production  

---

## üí° Quick Tip: IOC Hunting with EDR

EDR platforms allow analysts to search for **Indicators of Compromise (IOCs)** across all endpoints.

### Example:
- You identify a malicious file with MD5 hash:  
  `ac596d282e2f9b1501d66fce5a451f00`
- Search this hash in EDR
- Determine whether the file exists or is executing on other hosts

This helps identify:
- Scope of the attack  
- Additional compromised endpoints  

---

## üìå Conclusion

EDR is one of the most frequently used tools by SOC analysts, alongside **Log Management**.

Key takeaways:
- EDR provides deep endpoint visibility  
- Enables live investigations  
- Supports containment and IOC hunting  
- Effective use of EDR significantly improves investigation speed and accuracy  

Many analysts fail to fully leverage EDR capabilities. Mastering this tool gives you a strong advantage in SOC operations.

# ü§ñ SOAR ‚Äì Security Orchestration, Automation, and Response

**SOAR** stands for **Security Orchestration, Automation, and Response**. It enables different security tools and products within an environment to work together, significantly reducing the workload of **SOC team members**.

For example, a SOAR platform can automatically query **VirusTotal** for the source IP address of a SIEM alert, allowing the SOC analyst to focus on analysis rather than repetitive manual tasks.

---

## üîç What is SOAR?

SOAR platforms provide:
- Orchestration of multiple security tools  
- Automation of repetitive security tasks  
- Standardized incident response workflows  
- Centralized case and investigation management  

By automating common actions, SOAR improves efficiency, consistency, and response time in SOC operations.

---

## üß∞ Common SOAR Solutions

Some widely used SOAR products in the industry include:
- Splunk Phantom  
- IBM Resilient  
- Logsign  
- Demisto  

The image below demonstrates how SOAR can connect and automate multiple security tools within a single workflow.

![SOAR Capabilities](https://github.com/NATTOMR/SOC-Types-and-Roles/blob/main/SOAR-1.png)  
*Image source: hawk-eye.io*

---

## ‚è±Ô∏è Saves You Time

One of the biggest advantages of SOAR is **time savings through automation**.

Common automated workflows include:
- IP address reputation checks  
- File hash lookups  
- Scanning suspicious files in sandbox environments  
- Enriching alerts with third-party threat intelligence  

By automating these steps, analysts can focus on decision-making instead of repetitive lookups.

---

## üß© Centralization: One Platform for Everything

SOAR provides a **single platform** where multiple security tools are integrated, such as:
- Sandbox environments  
- Log Management systems  
- Threat Intelligence platforms  
- Third-party security tools  

This eliminates the need to switch between multiple dashboards during an investigation.

![SOAR Centralization](https://github.com/NATTOMR/SOC-Types-and-Roles/blob/main/SOAR-2.png)  
*Image source: splunk.com*

---

## üìã Playbooks

**Playbooks** are predefined workflows designed for specific incident scenarios.

Using SOAR playbooks, analysts can:
- Investigate SIEM alerts step by step  
- Follow standardized investigation procedures  
- Reduce mistakes caused by missed steps  

Even if an analyst does not remember every investigation step, playbooks guide them through the process.

---

### üîÑ Consistency Across the SOC Team

Playbooks also ensure **consistency** across all SOC team members.

For example:
- IP reputation checks should be performed by every analyst
- If one analyst skips this step, investigation quality suffers

By embedding this step into a playbook, all analysts follow the same investigation standard, improving overall SOC effectiveness.
